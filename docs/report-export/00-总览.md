# 报告导出功能 - 设计文档总览

> 版本: v1.0 | 日期: 2026-03-03 | 状态: 待Review

## 一、功能目标

在6个分析页面（IFIR×3 + RA×3）底部各添加一个"生成报告"按钮，点击后生成一份完整报告包含：

1. **Excel文件**：包含当前分析维度下的趋势数据、Top N排名、以及 Detail 明细原始数据
2. **图表图片**：使用后端代码根据原始数据程序化生成趋势图（非截图），嵌入Excel或单独导出

## 二、核心设计决策

### 2.1 图表生成方案：后端 Python matplotlib 生成

**理由：**
- 前端截图方案（html2canvas）对报告不友好：分辨率不可控、样式可能丢失
- 后端 matplotlib 可精确控制图表样式、字体、分辨率，输出高质量报告用图
- 图表可直接嵌入 Excel（openpyxl 支持插入图片），也可单独存为PNG

**技术栈：**
- `matplotlib` - 趋势折线图、饼图
- `openpyxl` - 生成多Sheet Excel + 嵌入图片
- FastAPI `StreamingResponse` - 文件下载流

### 2.2 整体架构

```
前端(按钮触发)
  → POST /api/{kpi}/report/{dimension}
    → 请求体 = 当前分析参数（不回传整批分析结果）
  → 后端接收参数
    → 复用现有 Service 查询逻辑（含 segment/segment2、ODM→plant 口径）
    → 生成 matplotlib 图表 → 内存PNG
    → 查询 Detail 明细数据（按需导出，受上限保护）
    → 构建 openpyxl Excel（多Sheet + 嵌入图表）
    → 先写入服务端临时文件，再返回文件下载响应
  → 前端下载文件
```

### 2.3 方案选择：后端重新查询 vs 前端传数据

选择 **后端重新查询**，理由：
- Detail 明细数据量大，前端不持有全量数据
- 后端已有完整的查询逻辑可复用
- 前端只需传筛选参数，不需要传大量数据体
- 保证报告数据与当前分析页面一致（使用相同参数）

**口径约束（必须遵守）**：
- 报告查询必须复用现有分析页 Service 中已经验证过的过滤逻辑
- `segment` 过滤必须兼容 `segment` / `segment2`
- `odms` 过滤必须复用现有动态 plant 解析能力，不能退回仅依赖 `map_odm_to_plant` 的静态映射
- 报告与页面展示的 Top N、Trend、Detail 必须基于同一套筛选口径

### 2.4 导出策略（v1.1）

采用 **同步导出 + 服务端临时文件**：
- 仍保留同步接口，便于前端直接下载
- Excel 先生成到服务端临时目录，再通过文件响应返回，避免在内存中同时保留 Workbook、PNG 和整份 `bytes`
- 明细数据导出设置上限，超过上限时在报告信息页和 Detail 页明确标记“已截断”
- 如果未来真实耗时稳定超过 60~120 秒，再升级为异步任务模式

## 五、Excel文件命名规范

```
{KPI}_{维度}分析报告_{实体名}_{时间范围}_{生成时间}.xlsx

示例：
IFIR_Model分析报告_ThinkPad-X1-Carbon_2024-07至2025-06_20260303.xlsx
RA_Segment分析报告_Consumer+Commercial_2024-01至2025-06_20260303.xlsx
```

## 七、中文图表约束

- 所有新增 Python / Markdown / 配置文件统一使用 **UTF-8 编码**
- 图表中的中文显示依赖“支持中文字形的字体文件”，不是仅靠 UTF-8 编码即可解决
- 报告导出模块应内置项目级中文字体文件（建议 `Noto Sans CJK SC` 或 `Source Han Sans SC`），统一用于 matplotlib 渲染
- 禁止依赖开发机或服务器操作系统上的硬编码字体路径作为唯一方案
